### 2. 你们为什么要抽象一个统一的 AgentRuntime？它解决了企业 IM 智能助手里的什么问题？

建议回答：

抽象 Runtime 的原因是，企业 IM 机器人不是一次性 Demo。不同客户会接不同 IM 平台、不同知识库、不同角色和工具，但底层会话、工具调用、上下文压缩、Skill 加载、异常处理其实是共性的。如果每个机器人单独开发，后续维护成本很高，同一个工具调用 bug 或上下文问题要在多个定制版本里重复修。

所以我们把 Agent loop、IM 消息接入、session 管理、context 压缩、tools 管理、skills 管理、system prompt 管理、记忆管理和 subagent 管理抽象成统一 Runtime。这样上层只需要根据客户场景配置角色、Skill 和入口适配，底层稳定性问题修一次可以复用到多个机器人。

它和普通聊天机器人最大的区别是：普通聊天机器人偏问答，而这个 Runtime 更像一个驻留在 IM 里的工作助手。它有长期运行的 session，有工具和文件能力，有后台任务，有角色和 Skill，也需要处理群聊、私聊、消息排队、失败恢复和本地化部署。

### 3. AgentRuntime 的会话管理是怎么做的？用户、群聊、平台之间如何隔离 session？

建议回答：

我们没有强行把不同平台的原始用户 ID 统一成一个全局 ID，而是统一抽象成 session key。比如飞书里群聊用 `group:${chatId}`，私聊用 `user:${senderId}`；CatsCompany 里用 `cc_group:${topic}` 或 `cc_user:${senderId}`；微信里用 `user:${fromUserId}`。这样不同平台的 ID 体系可以保持独立，但 Runtime 内部只关心标准化后的 session key。

每个 session key 对应一个独立的 `AgentSession`，里面有自己的 messages、上下文压缩状态、busy 状态和最近活跃时间。所以群聊和私聊上下文不共享，A 用户和 B 用户也不共享。共享的是 Runtime 级别的能力，比如同一套工具、Skill、模型配置和记忆服务。

会话过期通过 TTL 清理，默认大概是 60 分钟空闲后清理。清理时会做 session cleanup，把必要的对话摘要或记忆沉淀下来，并且可以判断是否需要主动唤醒用户。这样既能避免内存里长期挂很多空闲会话，也不会完全丢掉有价值的信息。

面试提醒：

你原来写“context 不共享，但是记忆共享”这个方向是对的，但要补清楚：共享的是服务能力或长期记忆，不是把不同用户的短期上下文混在一起。

### 4. subagent 任务调度模块是怎么设计的？父 Agent 和子 Agent 的上下文如何隔离？

建议回答：

subagent 主要解决长任务阻塞 IM 主会话的问题。主 Agent 调用 `spawn_subagent` 后，`SubAgentManager` 会生成一个 `sub-${uuid}` 的子任务 ID，并创建独立的 `SubAgentSession`。主会话不等待它执行完成，而是 fire-and-forget，让子任务在后台跑。

隔离上，子 Agent 有自己的 messages、system prompt、Skill 激活上下文和独立的 `ToolManager`。它的 sessionId 是 `subagent:${id}`，不会直接复用父会话 messages。父会话只把任务描述、用户原始指令、Skill 名称和工作目录传给子 Agent，避免把父会话完整上下文塞进去造成污染。

子 Agent 不直接给用户发消息，也没有直接的 IM channel。它和父会话通信主要通过 `notifyParent`、`ask_parent` 和最终完成通知。也就是说，子 Agent 可以把问题或结果注入给主 Agent，最终由主 Agent 判断怎么回复用户、要不要发送文件。

代码事实支撑：

- `SubAgentManager` 维护 `subAgents`、`parentMap` 和平台回调。
- `SubAgentSession` 内部维护独立 `messages`、`progressLog`、`resultSummary` 和 `outputFiles`。
- 子任务使用独立的 `ConversationRunner` 和 `ToolManager`，并设置 `sessionId: subagent:${id}`。

### 5. subagent 的状态机如何设计？running / completed / failed / stopped 分别如何流转？

建议回答：

subagent 的状态不是只靠一个异步 Promise，而是有明确生命周期。创建后默认是 `running`；如果执行成功，状态变成 `completed`，记录 `completedAt` 和 `resultSummary`；如果遇到不可重试错误，或者重试次数耗尽，就进入 `failed`；如果用户调用 `stop_subagent`，会设置 stopped 标志，状态变成 `stopped`，ConversationRunner 的 `shouldContinue` 会停止后续执行。

代码里还支持一个 `waiting_for_input` 状态。这个是子任务执行过程中遇到必须确认的问题时，通过 `ask_parent` 挂起当前子 Agent，并把问题注入给主会话。主 Agent 拿到用户或自己的判断后，可以用 `resume_subagent` 把答案传回去，子 Agent 再从 `waiting_for_input` 回到 `running`。

失败处理上，子任务内部有会话级重试，默认最多重试 2 次，基础延迟 5 秒，指数退避。可重试错误包括 429、rate limit、too many requests、overloaded、部分 5xx、529，以及 `ECONNRESET`、`ETIMEDOUT`、`ECONNABORTED` 这类网络错误。

进度日志通过 `progressLog` 保存，比如开始执行、重试、完成章节、生成 PPT、写出 summary 等。产出文件通过工具结果提取，比如 `write_file` 和 `pptx_generator` 的输出会被解析进 `outputFiles`，后续主 Agent 可以决定是否发送给用户。

### 6. subagent 执行完成后，结果是怎么回注到主会话的？

建议回答：

子任务完成后，`SubAgentManager` 会在 `session.run().finally()` 里拿到 `SubAgentInfo`，拼出一条类似“子智能体已完成/失败、任务是什么、结果摘要是什么、产出文件有哪些”的消息，然后通过平台回调 `injectMessage` 注入到父会话。

这里不是把子 Agent 的完整上下文追加回父会话，而是注入结果摘要和文件路径。这样能避免子任务内部大量工具调用、临时推理内容污染主会话。父 Agent 收到注入消息后，会像处理一条新消息一样进行推理，再决定是否回复用户、如何总结、是否调用 `send_file` 把产出发出去。

如果主会话当时正在 busy，飞书和 CatsCompany 适配层会等待并重试注入，避免子任务完成时刚好撞上主会话运行导致通知丢失。多个 subagent 同时完成时，没有强行做业务层排序，而是按完成时间注入；主会话本身有消息队列和 session busy 保护，保证不会并发写坏上下文。

### 7. Role / Skill 扩展体系是怎么实现的？

建议回答：

Role 可以理解为 Runtime 上的一层应用配置，它决定当前 Agent 的角色定位、system prompt、可继承的基础 Skill、要排除的 Skill，以及可注入的角色专属工具。每个 Role 有自己的 `role.json`、prompts、skills，必要时还有 role-specific tools。

Skill 是以目录为单位组织的，每个 Skill 包含一个 `SKILL.md`。系统通过 `PathResolver.findSkillFiles` 递归发现 `SKILL.md`，再用 `gray-matter` 解析 frontmatter。核心字段包括 `name`、`description`、`aliases`、`argument-hint`、`user-invocable`、`auto-invocable`、`max-turns`。其中 `description` 用于给 Agent 理解能力边界，`max-turns` 可以控制复杂 Skill 的工具调用轮次。

加载逻辑上，`SkillManager` 会先根据角色配置决定是否继承基础 skills，并根据 `excludeBaseSkills` 排除不需要的基础能力；再加载 `roles/<role>/skills` 下的角色 Skill。角色 Skill 可以通过同名覆盖基础 Skill。自动匹配策略比较保守，必须命中 Skill 名称或 alias，如果多个 Skill 命中，会优先选择名称更长、更具体的。

### 8. 你们沉淀了 20+ 垂直行业 Skills，能举一个具体例子吗？

建议回答：

可以举论文阅读或 LaTeX 编译相关 Skill。比如论文阅读 Skill 的输入通常是 PDF 路径、用户关注的问题或阅读目标；Skill 会先解析 PDF，必要时调用 MinerU 把 PDF 转成 Markdown、图片和表格，再结合 grep/read 等工具抽取关键段落，最后输出结构化阅读笔记、方法总结、实验设置和可追溯证据。

另一个例子是 LaTeX 编译 Skill。输入是论文项目目录或 tex 文件路径，输出是编译后的 PDF、错误定位和修复建议。它会把“检查依赖、运行编译、解析报错、定位 tex 行号、二次编译确认”这些重复流程沉淀成稳定工作流。

这些 Skill 的价值不是拍脑袋定义的，而是从真实使用中沉淀出来的。比如读论文、写论文、编译 PDF、画 Python 数据图、生信分析这些需求反复出现，手工流程稳定后再抽成 Skill。评估时主要看三点：能不能减少重复操作，输出是否可直接使用，失败时日志是否足够定位问题。

面试提醒：

“因为都是真实需求所以都有用”太弱。要说“真实需求 + 重复出现 + 流程稳定 + 输出可验收”。

### 9. 你们如何打通飞书、微信、CatsCompany 三类消息入口？

建议回答：

三类平台的接入方式不同，但进入 Runtime 前都会被适配成统一结构：消息文本、附件信息、chatType、senderId、sessionKey，以及用于回复和发文件的 `ChannelCallbacks`。

飞书是事件回调模式，解析 `im.message.receive_v1` 后，根据 `chat_type` 区分群聊和私聊，生成 `group:${chatId}` 或 `user:${senderId}`。CatsCompany 是 topic 模型，根据 topic 是否是群组生成 `cc_group:${topic}` 或 `cc_user:${senderId}`。微信是长轮询，收到消息后用 `from_user_id` 生成 `user:${from}`，同时保存 `context_token` 用于后续回复。

平台适配层只负责解析消息、下载附件、处理 slash command、构造 channel、处理 busy 时的消息队列，真正的推理都走统一的 `session.handleMessage(userText, { channel })`。这样 Runtime 不需要关心消息来自飞书还是微信，只需要面向统一的会话和通道接口。

群聊和私聊处理上，群聊有独立 session；飞书还支持 Bridge 广播，让多个 Bot 能在同一个群聊上下文里协作。附件处理则是先下载到本地，把文件路径和类型拼进用户消息，让 Agent 自己决定后续是否解析。

### 10. Agent 调用链路中的异常处理机制怎么做？

建议回答：

异常处理分两层：模型调用层和工具调用层。

模型调用层在 `AIService` 里做了统一封装。可重试 HTTP 状态码包括 408、429、500、502、503、504、529；网络层的 `ECONNRESET`、`ETIMEDOUT`、`ECONNABORTED`、`ENOTFOUND`、`EAI_AGAIN` 也会被识别为可重试。重试策略是最多 3 次，优先尊重 `Retry-After`，否则用 1 秒基础延迟的指数退避并加一点随机抖动，避免多个请求同时重试。

主备模型 fallback 也是在 `AIService` 层做的。系统会构建 provider chain，主模型失败后，如果错误属于可重试类型，或者是 401/403 这类配置级错误，并且有备模型配置，就会切换到下一个 endpoint。切换时会记录原 provider、model、目标 provider、model 和失败原因，方便排查。

工具调用层会把工具异常统一包装成 `ToolResult`，并标记 `ok=false`、`errorCode` 和 `retryable`。比如工具返回明确的 429、rate limit、too many requests、限流，就会标记为 `RATE_LIMIT` 且可重试。这样 ConversationRunner 可以根据工具结果决定是否继续推理或恢复。

### 11. Agent2Agent 协作链路是怎么设计的？HTTPBridge 承担什么职责？

建议回答：

Agent2Agent 协作主要通过 HTTPBridge 做 Bot 之间的通信。BridgeServer 提供了几个核心端点：`/group-message` 用于群聊广播，`/bot-message` 用于向另一个 Bot 派发任务，`/bot-result` 用于异步结果回传。BridgeClient 则负责向已配置的 peer Bot 发送任务、广播群聊消息或回传结果。

在群聊场景里，一个 Bot 回复后可以把自己的发言广播给其他 peer。其他 Bot 收到广播后，会把这条消息注入到同一个 group session 的上下文里。如果消息里 @ 了自己，就直接触发推理；如果没有 @，可以通过轻量的 chime-in judge 判断是否应该主动插话，并用随机短延迟避免多个 Agent 同时发言。

它和普通 tool call 的区别是：tool call 通常是主 Agent 调一个确定性的工具，结果直接回到当前推理链路；Agent2Agent 更像多个独立 Agent 在同一个群聊环境里协作，每个 Agent 有自己的角色、上下文、工具和输出策略。HTTPBridge 承担的是消息广播、任务投递、结果回传和基本认证，而不是直接替 Agent 做推理决策。

### 12. Electron 桌面端和可视化 Dashboard 做了哪些功能？

建议回答：

桌面端主要解决本地化部署和运维门槛问题。企业内部使用时，不可能每次都让用户手动跑命令、配环境、看终端日志，所以用 Electron 包了一层桌面端和 Dashboard。

Dashboard 里做了机器人服务管理，比如启动、停止、重启飞书/微信/CatsCompany 服务；也做了运行状态查看、日志查看、模型 API 切换、Role 切换、SkillsHub 下载、Skill 安装和卸载等功能。这样现场部署或试用时，可以通过界面完成大部分配置和排障。

## 二、GauzMem 长期记忆系统

### 13. GauzMem 为什么要设计“短期记忆 + 长期记忆”的双层架构？

建议回答：

短期记忆解决的是当前对话的即时上下文问题，比如最近几轮用户说了什么、当前任务进行到哪一步、有哪些临时文件或临时结论。长期记忆解决的是跨会话、跨时间的事实沉淀问题，比如人物偏好、事件关系、历史决策、时间线和因果链。

如果只靠短期上下文，长对话会受 token 限制影响，早期事实容易丢；如果所有信息都直接进长期记忆，又会引入大量噪声。所以我们设计成双层结构：短期记忆服务当前对话，长期记忆通过异步方式把有价值的内容沉淀成知识图谱和向量索引。

异步沉淀的原因是事实抽取、关系建模、向量写入、图谱更新都比较耗时，如果同步做会影响聊天响应速度。异步处理可以让前台对话保持流畅，后台再做结构化整理。

---

### 14. 为什么同时使用知识图谱和向量数据库？只用 RAG 不行吗？

建议回答：

向量检索擅长语义相似召回，比如用户问法和原文不完全一致时，也能找到相关 chunk。但向量检索对关系型问题不够强，比如“这个决定是因为什么导致的”“某个人前后态度有没有变化”“这件事之后又发生了什么”，这些需要沿着实体、事件、时间和因果关系做推理。

知识图谱擅长表达结构化关系，比如人物、事件、时间、因果、从属、共现等。它能把分散在多轮对话里的事实连接起来，支持多跳扩展和可解释检索。

所以两者结合后，向量负责先找到语义相关入口，图谱负责沿关系扩展上下文，最后再用时序信息补全事件前后关系。这样比单纯 RAG 更适合长期记忆和跨轮推理。

### 15. Facts 图和 Topics 图分别解决什么问题？

建议回答：

Facts 图解决细粒度事实和关系的问题。它把对话或文档中的内容抽成原子事实，比如“用户 A 负责项目 X”“项目 X 在 3 月延期”“延期原因是接口联调失败”。这些事实节点之间再建模因果、时序、从属、共现等关系，用来支持精确追溯和多跳推理。

Topics 图解决高层主题组织的问题。单个 fact 太碎，如果长期积累会变成大量孤立节点，不利于检索和解释。Topics 图会把相关 facts 聚合成主题，比如“项目延期”“接口联调”“用户偏好”“论文写作”等，让系统能够先定位主题，再下钻到具体事实。

早期可以用 Leiden 这类社区检测算法做主题聚合，后续也可以换成更轻量的相似度或连接密度聚类。面试里不用纠结算法名字，重点讲清楚：Facts 图负责原子事实，Topics 图负责把碎片事实组织成可解释主题。

---

### 16. “向量召回 + 图谱多跳扩展 + 时序上下文补全”的三级检索链路怎么工作？

建议回答：

用户问题进来后，系统先判断问题类型和检索策略。比如是问人物偏好、事件原因、时间线，还是普通事实。这个过程可以由 Agentic Search 完成，让 LLM 根据问题填写检索参数，比如检索实体、时间范围、关系类型、topK 和跳数。

第一步是向量召回，用问题 embedding 找到语义相关的 chunks 或 facts，得到一批候选入口。第二步是图谱多跳扩展，从候选事实出发，沿着实体、因果、时序、主题等边扩展相关节点。为了避免噪声扩大，会限制 hop 数、关系类型、分数阈值和每层扩展数量。

第三步是时序上下文补全。比如用户问一个事件的原因和后续影响，系统不仅召回事件本身，还会补齐它之前发生了什么、之后发生了什么。这样回答时可以形成时间线和因果链，而不是只给一个孤立片段。

---

### 17. 记忆代谢机制是怎么设计的？

建议回答：

记忆代谢的核心是给每条记忆一个价值分数，而不是长期无限存储。我们主要考虑时间衰减、召回次数和引用次数。时间衰减表示有些事实会随时间变旧；召回次数表示它经常被检索命中；引用次数表示它不仅被召回，而且真的被用于最终回答。

这里召回和引用要区分开。只被召回但很少被引用，说明它可能是噪声或泛化太宽的内容；经常被引用则说明它对回答有实际贡献。系统可以按加权公式计算记忆价值，对低价值记忆降权、归档或淘汰，控制长期记忆噪声和存储膨胀。

为了避免误删重要记忆，可以给用户明确偏好、身份信息、长期项目事实这类记忆更高初始权重，或者采用软删除/冷存储策略，先不参与热检索，但保留恢复能力。

### 18. LoCoMo 长期记忆基准实验是怎么做的？

建议回答：

LoCoMo 主要考察长期对话记忆能力，问题会覆盖单跳事实、多跳推理、时间线和跨轮信息整合。实验流程是先把长期对话数据写入记忆系统，再对 benchmark 问题进行检索和回答，最后用 LLM Judge 对回答和参考答案进行一致性判断。

LLMJudge 准确率可以理解为对每道题给 0/1 或等价评分，然后统计正确比例。Single-hop 更偏单个事实的召回，比如某个人说过什么；Multi-hop 需要把多个事实连接起来，比如一个事件的原因、后续影响或人物关系变化。

我们复现实验里总体 LLMJudge 准确率是 86.75%，其中 Multi-hop 89.4%、Single-hop 87.9%。这个结果说明“向量召回 + 图谱扩展 + 时序补全”的设计对长期记忆和多跳问题有帮助。但面试时不要直接说 SOTA，除非你准备好了完整 baseline 和对比表。

## 三、VibeWritingPlatform 文档写作平台

### 19. VibeWritingPlatform 主要解决长文档生成中的哪些问题？

建议回答：

它主要解决三个问题：结构不稳定、内容不可追溯、后编辑成本高。

结构不稳定指的是模型生成长文档时容易偏离大纲，比如章节粒度不一致、标题风格不一致、前后重复、某些章节过长或过短。内容不可追溯指的是生成内容不知道来自哪份资料、哪一页、哪张图表，容易产生幻觉，也不方便用户审查。后编辑成本高则是生成完以后用户还要自己改局部段落、调整语气、补图表、修逻辑。

所以平台把长文档写作拆成主对话、资料管理、模板解析、章节规划、分章节生成、Review 和后编辑链路。目标不是一次性生成一篇完美文章，而是让长文档生成过程可控、可追溯、可修改。

---

### 20. 你们为什么把主对话、RAG、Review、Editor、Python 图表生成拆成独立服务？

建议回答：

主要是为了降低耦合和方便独立迭代。主对话 Agent 负责理解用户意图和协调流程；资料/模板 RAG 负责解析和召回证据；Review 负责检查连贯性和逻辑一致性；Editor 负责局部修改；Python 图表服务负责生成和执行图表代码。这些模块的依赖、运行耗时和失败模式都不一样，拆开后更容易维护。

拆服务还有一个好处是故障隔离。比如 Python 图表服务挂了，用户仍然可以继续写文档和做文本后编辑；RAG 服务暂时不可用，也可以提示用户重新上传资料或先基于模板生成草稿；Review 服务失败，不影响 Editor 做局部改写。

HTTP API 会增加一些接口定义和错误处理成本，但只要请求/响应 schema 清楚，复杂度是可控的。相比所有逻辑塞进一个大 Agent，模块化更适合长文档这种多阶段流程。

---

### 21. Plan-and-Execute 多 Agent 流程是怎么落地的？

建议回答：

主对话 Agent 负责理解用户目标，比如要写什么类型的文档、面向什么场景、使用哪些资料和模板。Planner 负责解析模板和资料，生成章节规划、写作顺序、每章要点、风格要求和证据需求。Executor 根据 Planner 的计划分章节生成内容，并在每个章节里做 RAG 召回。Review 最后检查章节之间是否重复、矛盾、跳跃，整体逻辑是否连贯。

Executor 分章节并行时，风格一致性主要靠共享写作计划、统一 prompt 约束、模板结构和术语表来保证。每个章节不是自由发挥，而是围绕 Planner 给出的章节目标、证据范围和写作风格执行。

Review 判断逻辑一致性时，不只是看单章内容，而是看章节头尾衔接、概念是否重复定义、结论是否被前文支撑、不同章节是否出现冲突表述。你原来说“看头尾”可以保留，但面试时要说成“检查章节边界和全文结构连贯性”。

---

### 22. MinerU + RAG 的可追溯资料链路怎么做？

建议回答：

MinerU 负责把 PDF 等资料解析成 Markdown、图片、表格等结构化内容。解析后会把文本按章节、页码、标题层级、图表编号等信息切成 chunks，并在 metadata 里保存来源文件、页码、章节、元素类型、图片路径等信息。

RAG 检索时不仅返回文本内容，也返回 metadata。Executor 生成内容时可以把证据来源一起带入 prompt，要求模型基于证据写作，并在生成结果里保留引用位置。这样用户看到某段生成内容时，可以追溯到原始资料的文件、页码或图表。

这个链路的价值是降低幻觉和审查成本。长文档不是只要写得像，还要能解释“这句话从哪里来”。

---

### 23. 类 Cursor 的文档后编辑体验具体有哪些功能？

建议回答：

核心是选中文本后做局部 AI 编辑。用户在 Monaco Editor 里选中一段内容，输入“改得更正式”“扩写这一段”“压缩成一段话”这类指令。系统会把选区内容、附近上下文和用户指令发给模型，模型返回替换文本或 diff，再在编辑器里展示给用户确认。

控制范围主要靠选区边界和上下文窗口。模型可以看到附近上下文来保持连贯，但只能修改用户选中的区域，避免全篇被意外改动。语气调整、局部扩写、润色都可以用这个模式实现。

Python 图表生成则是用户用自然语言描述图表需求，模型生成 Python 代码，在沙盒或受限环境里执行，产出图片后插入文档。这里重点是安全和可控，比如限制执行目录、设置超时、保存输出文件，并把生成的图表路径插回文档。

如果提到图像生成服务，可以说“也接过图像生成能力用于插图”，不要在面试里说太多外号。

## 四、综合技术能力

### 24. 如果让你给公司内部做一个企业知识助手，你会怎么设计整体架构？

建议回答：

我会分成数据接入层、知识处理层、检索层、Agent 编排层、权限层和评测运维层。

数据接入层负责接入企业文档、飞书/企业微信消息、知识库、工单、代码仓库等来源。知识处理层负责解析文档、OCR/表格/图片抽取、chunk 切分、metadata 标注和增量更新。检索层会同时做向量索引和关键词索引，必要时对实体、项目、人员和事件做图谱建模。

权限是企业知识助手最关键的一层。每个 chunk 和文档都要带 ACL metadata，检索时先根据用户身份、部门、群组权限过滤，再进入 rerank 和生成，不能只在回答后做过滤。

Agent 层负责把 RAG 和工具结合起来。简单问题走检索问答；复杂问题由 Agent 分解任务，比如查资料、比较版本、生成报告、发起审批或调用内部系统。为了降低成本，可以做 query rewrite、分层召回、摘要缓存、历史上下文压缩和小模型分类，把大模型调用留给真正需要推理的步骤。

评估上，我会建立离线黄金问答集和线上反馈闭环，关注准确率、引用命中率、拒答正确率、权限越权率、延迟和 token 成本。企业场景里，不答错比答得花哨更重要。

### 25. 你如何理解 Agent 和传统 RAG 应用的区别？

建议回答：

传统 RAG 更像“检索 + 生成”的问答系统，流程通常是用户问题进来，召回相关资料，再让模型基于资料回答。它适合事实问答，但流程比较固定。

Agent 的核心是能根据目标动态规划和行动。它不只是检索资料，还可以选择工具、分解任务、执行多步操作、检查中间结果、在失败后重试或换策略。比如写报告时，Agent 可以先解析模板，再检索资料，再分章节生成，再 Review，再调用编辑器或图表工具。

Agent 系统最大的工程难点不是“让模型会说话”，而是可控性和稳定性：上下文怎么管理，工具权限怎么限制，失败怎么恢复，长任务怎么不中断主会话，成本和延迟怎么控制，最终效果怎么评估。

### 26. 你做过的项目里，最难 debug 的一次问题是什么？最后怎么定位和解决的？

建议回答：

可以讲 subagent 结果回注的问题。

背景是在 IM 场景里，用户派了一个长任务给 subagent，子任务后台确实完成了，但主会话有时没有及时把结果告诉用户。现象是日志里能看到子任务 completed，但用户侧没有收到最终总结或文件。

我当时先从日志链路排查：确认子任务是否创建、状态是否从 running 到 completed、resultSummary 和 outputFiles 是否生成；然后继续查 SubAgentManager 的完成回调，发现问题集中在“子任务完成时主会话可能正在 busy”，这时直接注入会和主会话当前推理产生竞态。

解决方案是把平台回调持久注册到 SubAgentManager，子任务完成后不直接发用户，而是通过 `injectMessage` 注入主会话；平台适配层在注入时检查 session busy，如果忙就等待并重试，最多重试多次。同时子任务完成信息会保留一段时间，用户也可以通过 `check_subagent` 查询。

复盘来看，这个问题让我意识到 IM Agent 和 CLI Agent 的不同：IM 场景里主会话必须一直可响应，后台任务结果、用户新消息和工具执行会并发出现，所以必须把 session busy、消息队列和异步结果回注作为 Runtime 基础能力设计。

我的优势是做过比较完整的 Agent 工程化链路，不只是调用模型 API。我参与过 Runtime 底座、会话隔离、工具调用、Role/Skill 扩展、subagent 调度、多平台 IM 接入、异常重试和模型 fallback，也做过 RAG、知识图谱和长期记忆。

## 六、高频题完整回答

### 高频题 1：请详细讲一下 subagent 调度模块

我的完整回答：

subagent 调度模块主要是为了解决 IM Agent 里的长任务阻塞问题。比如用户让 Agent 精读论文、生成报告、跑复杂分析，如果主会话一直同步执行，用户就只能等，而且中间无法继续交流。所以我们设计了 subagent，让主 Agent 像调度器一样把长任务派给后台子 Agent。

实现上，主 Agent 调用 `spawn_subagent`，传入 skill_name、task_description 和完整用户指令。`SubAgentManager` 会检查当前父会话下正在运行的子任务数量，默认最多 3 个；然后创建 `sub-${uuid}` 的 ID，建立 subagent 到 parentSessionKey 的映射，并创建独立的 `SubAgentSession`。

子 Agent 和父 Agent 的隔离比较明确。子 Agent 不复用父会话 messages，而是重新构造 system prompt，激活指定 Skill，把用户指令作为当前任务输入，并创建独立的 ConversationRunner 和 ToolManager。它的 sessionId 是 `subagent:${id}`，工具权限也按 strict profile 运行。这样子任务内部的大量工具调用和临时推理不会污染父会话。

状态上，子任务创建后是 running，成功后 completed，失败后 failed，用户停止后 stopped。如果子任务中途需要确认，可以调用 `ask_parent` 进入 `waiting_for_input`，主会话收到问题后再用 `resume_subagent` 恢复。失败时会识别限流、部分 5xx、网络错误等可重试错误，做会话级指数退避重试。

结果回注上，子任务完成后不会直接发给用户，而是通过平台回调把“任务完成/失败、结果摘要、产出文件列表”注入主会话。主 Agent 再决定如何总结给用户、是否发送文件。这保证了主会话仍然是唯一对用户负责的出口，也让后台任务和 IM 对话保持一致。

### 高频题 2：为什么要用知识图谱做长期记忆

我的完整回答：

长期记忆和普通文档 RAG 不太一样。RAG 更擅长从文档里找语义相关片段，但长期对话记忆里很多问题不是“哪段话相似”，而是“某件事为什么发生”“某个人和某项目有什么关系”“之前的决定后来有没有变化”。这些问题需要关系和时间线。

知识图谱的价值是把分散在多轮对话里的事实结构化。比如从对话中抽取出人物、项目、事件、时间、原因等事实节点，再建立因果、时序、从属、共现等边。这样检索时不只是找到一个 chunk，而是可以沿着实体和关系做多跳扩展，补出上下游事实。

我们不是用图谱替代向量，而是结合使用。向量负责语义召回，找到可能相关的入口；图谱负责关系扩展，找出与入口相关的人物、事件和因果链；时序补全负责把事件前后文补齐。这样能同时兼顾模糊问法、结构化推理和可解释性。

### 高频题 3：如果做一个公司内部知识助手，你会怎么设计

我的完整回答：

我会先把它当成企业级系统，而不是普通聊天机器人来设计。整体分为数据接入、知识处理、检索、Agent 编排、权限控制和评测运维。

数据接入层对接企业文档、知识库、IM 群消息、工单、代码仓库等来源。知识处理层负责解析文件、抽取文本/表格/图片、切 chunk、打 metadata，并做增量更新。检索层用向量召回加关键词检索，重要业务实体可以建图谱。每个 chunk 都要带来源、时间、部门、权限、版本等 metadata。

权限控制必须前置。用户提问后，检索阶段就要根据用户身份和 ACL 过滤不可见内容，不能等模型回答完再过滤。Agent 层则根据问题复杂度选择路线：简单问题直接 RAG；复杂问题做任务分解，调用搜索、数据库、工单、文档生成等工具。

最后要有评测和观测。离线用黄金问答集测准确率、引用正确率和拒答能力；线上记录用户反馈、无答案率、延迟、token 成本和权限风险。企业助手的核心不是回答得热闹，而是准确、可追溯、可控、不会越权。

### 高频题 4：你项目里最有技术含量的部分是什么

我的完整回答：

我觉得最有技术含量的是 AgentRuntime 里的 subagent 调度和 Role/Skill 扩展体系。因为这两个模块都在解决 Agent 从 Demo 到真实可用系统的问题。

subagent 调度解决的是长任务和主会话响应性的矛盾。它不是简单开一个异步任务，而是要处理父子会话隔离、状态机、进度日志、产出文件、用户停止、失败重试、等待父会话输入和最终结果回注。如果这些细节没处理好，IM 场景里很容易出现任务丢失、上下文污染或用户不知道任务状态的问题。

Role/Skill 体系解决的是可复用和可定制问题。不同企业、不同场景不可能每次重新写一个 Agent，所以我们把角色 prompt、可用工具、基础 Skill、角色 Skill、Skill 元数据和自动匹配机制抽象出来。这样 Runtime 底座稳定后，上层可以通过配置和 Skill 快速适配新行业。

这两个模块结合起来，让 Agent 不只是一个聊天入口，而是一个能在企业 IM 中长期运行、可扩展、可维护的智能助手底座。

### 高频题 5：你遇到过最难的问题是什么

我的完整回答：

我可以讲一个 subagent 完成结果偶发丢失的问题。这个问题难在它不是稳定复现的代码异常，而是异步时序问题。

当时用户在 IM 里派发了一个后台任务，subagent 日志里显示已经完成，也生成了结果摘要和文件路径，但用户侧偶尔没有收到最终通知。我先查子任务状态流，确认 running 到 completed 是正常的；再查文件产出，确认 outputFiles 也有；最后定位到完成通知回注主会话时，主 session 可能正处在 busy 状态，导致注入和当前推理发生竞态。

修复思路是把子任务完成后的通知统一走平台层的 `injectMessage`，并在平台适配层加 busy 检查和重试。主会话空闲后再把子任务结果作为一条新消息交给 Agent 处理。同时保留 completed 状态一段时间，用户可以通过 check_subagent 手动查询。

这个问题给我的收获是，Agent 工程里很多 bug 不在模型本身，而在 Runtime 的状态管理和异步链路。尤其是 IM 场景，用户消息、后台任务、工具调用和平台回调会交织在一起，必须有清晰的 session 状态、队列和日志。
