# Experiment Runner Doctor

规范地启动长实验，跟踪状态，并在失败时给出诊断，而不是靠临场 shell 硬试。

## 触发条件

- "跑一下这个实验"
- "帮我盯着这个训练"
- "看看这个实验为什么挂了"
- "恢复这个实验"

## 硬规则

1. 启动实验前先确认工作目录、入口脚本和输出日志位置
2. 长实验必须有日志或状态输出，不要裸跑后什么都拿不到
3. 如果出现编码、权限、路径、环境问题，要先归因为运行环境问题
4. 失败后优先输出诊断，不要无限重试

## 工作流程

1. 确认实验入口、依赖环境和目标日志文件
2. 生成规范化运行命令
3. 启动实验并记录状态
4. 用日志或输出检查：
   - 是否真正启动
   - 是否卡死
   - 是否报错
5. 输出：
   - 当前状态
   - 日志位置
   - 错误摘要
   - 下一步建议

## 输出要求

- 实验是否已启动
- 当前运行到哪
- 是否存在超时、编码或依赖问题
- 是否建议继续、停止或调整命令

## 注意事项

- 不要把“运行时间长”误判成“已经卡死”
- 如果只是日志太少，要明确说“证据不足”
- 如果实验属于高成本任务，先把命令和风险讲清楚
