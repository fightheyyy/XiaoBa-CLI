# WebCLI 优化方案

## 核心问题

当前 skill 让 agent 执行脚本，脚本**被动等待**网络请求。
用户不点击，就发现不了 API。覆盖率低，不够智能。

## 目标

agent 自己操作浏览器，主动触发所有可交互元素，系统性发现全部 API。

---

## 优化方向一：Agent 主动操作浏览器（核心改进）

### 现在的流程
```
用户说"CLI化" → agent 执行脚本 → 脚本被动监听 → 用户手动点击 → 发现部分API
```

### 优化后的流程
```
用户说"CLI化" → agent 打开页面 → agent 扫描所有可交互元素 → agent 逐个点击 → 捕获所有API
```

### SKILL.md 需要新增的指令

给 agent 明确的操作步骤：

1. 打开目标页面，等待加载完成
2. 执行 `document.querySelectorAll('button, a[href], [onclick], form, [data-action]')` 获取所有可交互元素
3. 过滤掉：外链、纯样式按钮、登出按钮等危险操作
4. 对每个元素：记录文本/属性 → 点击/提交 → 等待网络请求 → 记录捕获的 API
5. 处理 SPA 路由变化：点击后如果 URL 变了，返回上一页继续
6. 汇总所有 API，去重，生成 SKILL.md

---

## 优化方向二：分层探索

不是无脑点所有元素，而是按层级探索：

```
首页
├── 导航菜单（一级）→ 点击进入各模块
│   ├── 模块页面（二级）→ 扫描该页面的按钮/表单
│   │   └── 弹窗/详情（三级）→ 捕获详情API
```

避免遗漏隐藏在子页面里的 API。

---

## 优化方向三：智能过滤危险操作

点击前先判断元素是否危险：

```javascript
const dangerous = ['删除', '清空', '注销', '退出', 'delete', 'logout', 'remove']
// 跳过包含这些关键词的按钮
```

---

## 优化方向四：表单参数自动识别

遇到表单时，不只是提交空表单，而是：
1. 分析 `<input>` 的 `name`/`placeholder`/`type`
2. 自动填入测试值（搜索框填"test"，日期填今天，数字填1）
3. 提交，捕获带参数的 API 请求
4. 记录参数名和类型

## 参考：OpenCLI 的做法

OpenCLI 的 `record` workflow 也是被动的（用户手动操作），
但它的 SKILL.md 明确告诉 agent：**"你要用浏览器工具主动去点"**。

我们的 SKILL.md 缺少这个关键指令。最小改动是：
**在 SKILL.md 里加一段"Agent 操作指南"，告诉 agent 要主动点击页面元素。**
